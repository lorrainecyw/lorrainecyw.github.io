    18 | StampedLock：比读写锁性能更好

    lorraine 2022-07-05

    # StampedLock 支持的 3 种锁模式

    针对读多写少的场景，Java 在 1.8 版本中提供了StampedLock，比ReadWriteLock性能更好 StampedLock提供了 3 种模式：写锁、悲观读锁、乐观读，比ReadWriteLock多了乐观读

    1. 写锁：与ReadWriteLock中的写锁相似，只允许一个线程获取写锁
    2. 悲观读锁：与ReadWriteLock中的读锁相似，允许多个线程获取悲观读锁
    3. 乐观读：无锁的操作，允许多个线程获取乐观读和读锁，允许一个线程获取写锁。

    补充点：

    • 写锁和悲观读锁是互斥的，且当获取写锁或者获取悲观读锁成功后，会返回一个 long 类型的 stamp，在解锁的时候需要传入这个 stamp
    • 乐观读允许一个写线程获取写锁，所以不会导致所有写线程阻塞，也就是当读多写少的时候，写线程有机会获取写锁，减少了线程饥饿的问题，吞吐量大大提高；而且由于乐观读并不是锁，所以没有线程唤醒与阻塞导致的上下文切换，性能更好

    # 获取写锁 / 悲观读锁

    final StampedLock sl = new StampedLock();

// 获取 / 释放悲观读锁示意代码
long stamp = sl.readLock();
try {
	// 省略业务相关代码
} finally {
	sl.unlockRead(stamp);
}

// 获取 / 释放写锁示意代码
long stamp = sl.writeLock();
try {
	// 省略业务相关代码
} finally {
	sl.unlockWrite(stamp);
}

    # 乐观读

    将共享变量读取到局部变量之后，需要通过lock.validate(stamp)校验在执行读操作的过程中是否被写线程修改过，如果修改过则需要升级到悲观读锁，再重新读取共享变量到局部变量

    class Point {
	private int x, y;
	final StampedLock sl = new StampedLock();
	// 计算到原点的距离
	int distanceFromOrigin() {
		// 乐观读
		long stamp = sl.tryOptimisticRead();
		// 读入局部变量，在读的过程数据可能被修改
		int curX = x, curY = y;
		// 判断执行读操作期间，是否存在写操作，
		// 如果存在，则 sl.validate 返回 false
		if (!sl.validate(stamp)){
			// 升级为悲观读锁
			stamp = sl.readLock();
			try {
				curX = x;
				curY = y;
			} finally {
				// 释放悲观读锁
				sl.unlockRead(stamp);
			}
		}
		return Math.sqrt(curX * curX + curY * curY);
	}
}

    # 锁升级与锁降级

    StampedLock支持锁的降级和升级

    1. 降级：通过tryConvertToReadLock(long stamp)方法实现，会返回一个 stamp
    2. 升级：通过tryConvertToWriteLock(long stamp)方法实现，会返回一个 stamp

    # StampedLock 使用的注意事项

    StampedLock在简单的应用场景基本能够替代ReadWriteLock，但是拥有的功能仅仅是ReadWriteLock所拥有的子集

    1. StampedLock是不可重入锁
    2. StampedLock的写锁、悲观读锁都不支持条件变量
    3. 如果线程阻塞在StampedLock的readLock()或者writeLock()上时，此时调用该阻塞线程的 interrupt()方法，会导致 CPU 飙升。所以如果支持中断功能，一开始获取锁时要采用悲观读锁的readLockInterruptibly()和写锁的writeLockInterruptibly()
