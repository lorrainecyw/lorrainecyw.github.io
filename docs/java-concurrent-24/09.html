    09 | Java 线程（上）：Java 线程的生命周期

    lorraine 2022-07-02 线程

    在 Java 中，实现并发程序的主要手段就是多线程

    # 通用的线程生命周期

    通用的线程声明周期可以用“五态模型”来描述，五种状态分别是：初始状态（创建）、可运行状态（就绪）、运行状态、休眠状态、终止状态

    1. 初始状态：线程已经被创建，但是还不允许分配 CPU 执行。这里所谓的被创建，指的是在编程语言层面被创建，在操作系统层面线程还没有创建
    2. 可运行状态：可以分配 CPU 给线程来执行了。这个状态下，真正的操作系统线程已经创建好了
    3. 运行状态：当有空闲的 CPU 时，操作系统将 CPU 分配给一个处于可运行状态的线程，线程得以运行
    4. 休眠状态：当处于运行状态的线程调用了阻塞式 API 或者等待某个条件变量（ I / O 设备等），就会转换到休眠状态，并且释放 CPU 使用权。等到等待的事件出现了，才会重新回到可运行状态
    5. 终止状态：线程正常执行完或者抛出异常就会进入终止状态，意味着生命周期的结束

    # Java 中线程的生命周期

    # 生命周期包含的状态

    Java 中线程共有六种状态，分别为：

    1. NEW：初始化状态
    2. RUNNABLE：可运行 / 运行状态（包含了这两种状态）
    3. WAITING：等待状态
    4. TIMED_WAITING：有时限等待状态（等待达到了一定时间就会转变为RUNNABLE）
    5. BLOCKED：阻塞状态
    6. TERMINATED：终止状态

    注意点：

    • WAITING、TIMED_WAITING、BLOCKED都属于前面提到的睡眠状态，也就是说，只要 Java 线程处于这三种状态之一，就没有对 CPU 的使用权
    • Java 语言里则把可运行状态和运行状态合并了，这两个状态在操作系统调度层面有用，而 JVM 把线程调度交给操作系统处理了，所以 JVM 层面不关心这两个状态，在JVM 看来，等待CPU 使用权（操作系统层面此时处于可执行状态）与等待 I/O（操作系统层面此时处于休眠状态）没有区别，都是在等待某个资源，所以都归入了 RUNNABLE 状态

    # 状态转换

    # NEW ==> RUNNABLE

    Java 中刚创建好的线程就是NEW状态，创建线程有 4 种方法：继承 Thread 类，实现 Runnable 接口，实现 Callable 接口，通过线程池创建。下面用代码演示最常用的两种：

    /** 
*	1. 继承Thread，重写run方法
*/

class MyThread extends Thread {
	public void run() {
		//线程需要执行的代码
	}
}
//创建线程对象
MyThread myThread = new MyThread();


/** 
*	2. 实现Runnable接口，重写run方法
*/

class Runner implements Runnable {
	@Override
	public void run() {
		//线程需要执行的代码
	}
}
//创建线程对象
Thread thread = new Thread(new Runner());

    从NEW状态转换到RUNNABLE状态，只需要调用线程对象的start()就好了，例如myThread.start();

    # RUNNABLE <==> BLOCKED

    只有一种情况会使得线程从RUNNABLE转换到BLOCKED，那就是线程等待synchronized的隐式锁的时候。synchronized修饰的方法和代码块在同一时刻只能有一个线程访问，那么没有获得锁的线程就会阻塞等待，从RUNNABLE转换到BLOCKED，等到获得隐式锁之后又会从BLOKED转换到RUNNABLE 注意：如果调用阻塞式 API，在操作系统层面，会转换到休眠状态，但是在 JVM 层面，不会发生状态的转换，所以我们平时说的调用阻塞式 API，线程会阻塞等待，指的是操作系统层面线程的状态

    # RUNNABLE <==> WAITING

    总的来说，有 3 种场景会触发这种转换

    1. 调用wait()：获得了synchronized隐式锁的线程调用wait()，释放锁，状态发生转换
    2. 调用join()：当调用了thread1.join()，当前线程就会从RUNNABLE转换到WAITING，直到thread1执行完后，当前线程才会从WAITING转换到RUNNABLE，继续执行
    3. 调用LockSupport.park()：Java 并发包的锁都是基于LockSupport实现的，调用LockSupport.park()，当前线程会阻塞，转换到WAITING状态；调用LockSupport.unpark(Thread thread)，会唤醒目标线程，转换回RUNNABLE状态

    # RUNNABLE <==> TIMED_WAITING

    有 5 种场景会触发这种转换（调用之后，状态：RUNNABLE => TIMED_WAITING，时间到了就会转回来）

    1. 调用带超时参数的Thread.sleep(long millis)
    2. 调用带超时参数的Object.wait(long timeout)（获得synchronized隐式锁后才能调用）
    3. 调用带超时参数的Thread.join(long millis)
    4. 调用带超时参数的LockSupport.parkNanos(Object blocker, long deadline)
    5. 调用带超时参数的LockSupport.parkUntil(long deadline)

    # RUNNABLE ==> TERMINATED

    1. 正常执行完run()后
    2. 抛出异常后
    3. 调用stop()（已被标记为@Deprecated，不建议使用）
    4. 调用interrupt()
    # 为什么不建议使用stop()

    调用stop()后，会不给线程喘息的机会，直接杀死线程，如果线程持有ReentrantLock等锁，被stop()的线程并不会自动调用unlock()方法来释放这个锁，那么就导致其他线程也无法获得这个锁，实在是太危险了（如果持有的是synchronized锁是会释放的，JVM 会释放）。类似的还有suspend()和resume()，同样不建议使用

    # 调用interrupt()

    interrupt()仅仅是通知线程，线程有机会执行后续的一些操作，也可以无视这个通知 而收到通知的方式有两种，一种是异常，一种是主动监测

    1. 异常
      1. 当线程处于WAITING或者TIMED_WAITING状态的时候，如果调用了interrupt()，线程就会转换到RUNNABLE状态，并且触发InterruptedException异常（这个异常的触发条件：其他线程调用了这个线程的interrupt()）
      2. 当线程处于RUNNABLE状态时，并且阻塞在java.nio.channels.InterruptibleChannel上时，如果其他线程调用线程 A 的interrupt()，线程 A 会触发java.nio.channels.ClosedByInterruptException这个异常；而阻塞在java.nio.channels.Selector上时，如果其他线程调用线程 A 的 interrupt()，线程 A 的java.nio.channels.Selector会立即返回
    2. 主动监测：当线程处于RUNNABLE状态时，如果其他线程调用了当前线程的interrupt()可以通过isInterrupted()检测是不是自己被中断了

    注意：抛出异常后，会自动清除当前线程的中断标识

    # 课后思考

    下面代码的本意是当前线程被中断之后，退出while(true)，你觉得这段代码是否正确呢 ？

    Thread th = Thread.currentThread();
while(true) {
	if(th.isInterrupted()) {
		break;
	}
	// 省略业务代码无数
	try {
		Thread.sleep(100);
	} catch (InterruptedException e){
		e.printStackTrace();
	}
}

    分析如下

    1. 结果：可能会陷入无限循环
    2. 原因：线程在执行Thread.sleep(100)的期间的状态为TIMED_WAITING，如果在被打断了，就会抛出InterruptedException异常，抛出异常后，会自动清除掉中断标识，这样子th.isInterrupted()得到的结果永远是false，没有办法执行break出循环这一步
    3. 解决：可以在catch到异常的代码块中重置一下中断标识（具体做法：th.interrupt();），这样th.isInterrupted()就能够得到true的结果并执行break；也可以再捕获到异常后直接在catch里面执行break
