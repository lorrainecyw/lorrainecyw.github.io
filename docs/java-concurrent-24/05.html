    05 | 如何预防死锁

    lorraine 2022-07-03

    上一讲中，提到了银行转账的问题，并针对这个问题提出了解决方案，但是使用Account.class作为互斥锁极大地降低了性能，所以应该如何优化呢 ？ 从分析上看，账户 A 向账户 B 转账 100 元，那么 A 的账户余额减少 100 元，B 的账户余额增加 100 元。这其中涉及了两个步骤，既要锁住 A 的账户余额，又要锁住 B 的账户余额。账户 A 是转出账户，账户 B 是转入账户 可以想到，不一定要使用同一把锁来同时锁住 A 和 B，可以用第一把锁来锁住 A，第二把锁来锁住 B，实现如下：

    class Account {
	//账户余额
	private int balance;
	//转账，当前账户 -> target账户
	void transfer(Account target, int money) {
		//锁住转出账户
		synchronized(this) {
			//锁住转入账户
			synchronized(target) {
				if(this.balance > money) {
					this.balance -= money;
					target.balance += money;
				}
			}
		}
	}
}

    看上去没有问题，但实际却有很大的漏洞，假设线程 1 执行 A 向 B 转账，与此同时线程 2 执行 B 向 A 转账，那么线程 1 获取到了账户 A 的锁，线程 2 获取到了账户 B 的锁，接下来线程 1 要获取账户 B 的锁时发现获取不到了，因为此时账户 B 的锁被线程 2 所占有，同样的，线程 2 也获取不到账户 A 的锁，两个线程就只能一直等待下去。这就叫死锁

    # 如何预防死锁

    死锁：两个或多个线程因为竞争资源而导致相互等待，导致一直阻塞下去的现象

    # 造成死锁的 4 个条件

    这 4 个条件必须都满足才会造成死锁

    1. 互斥：该共享资源在同一时刻只能被一个线程所占有
    2. 占有并等待：线程占有某个或者某些共享资源，并且等待其他的共享资源
    3. 不可剥夺：其他线程不能抢占当前线程所占有的共享资源，除非当前线程自己释放
    4. 循环等待：多个线程等待资源的顺序形成了环路，例如 A 等待 B 的资源，B 等待 C 的资源，C 等待 A 的资源

    # 如何避免死锁发生

    针对上面的 4 个条件，由于必须要 4 个条件都满足才会导致死锁，所以只要破坏其中一个条件就可以避免死锁 对于“互斥”这个条件是没有办法破坏的，因为使用锁就是为了互斥，针对其他三个条件，可以这样破坏：

    1. 对于“占有并等待”，可以一次性申请所有需要的资源，这样就不需要等待了
    2. 对于“不可剥夺”，可以让线程在申请不到其他资源的时候，主动释放自己占有的资源
    3. 对于“循环等待”，可以给资源标序号，按照顺序从小到大申请资源，这样就不存在循环了

    # 破坏“占有并等待”条件

    既然要一次性申请所有需要的资源，那么如果其中有的资源申请不到，那就应该等会再申请（通过循环实现不断申请）。应该向谁申请呢？应该有一个类似管理员的类，这个管理员要保存当前已被占用的资源列表，这样才能判断资源能不能被申请走，要有申请资源和释放资源的功能 具体实现如下：（以银行转账为例，有转出账户和转入账户）

    //管理员
class Administrator {
	//存放已经被占用的资源
	private List<Object> sourses = new ArrayList<>();
	synchronized boolean apply(Object from, Object to) {
		//如果 from 或者 to 有一个被占用了，那就无法申请到
		if(sources.contains(from) || sources.contains(to)) {
			return false;
		} else {
			sources.add(from);
			sources.add(to);
			return true;
		}
	}
	synchronized void free(Object from, Object to) {
		sources.remove(from);
		sources.remove(to);
	}
}
//账户
class Account {
	//admin 应该为单例
	private Administrator admin;
	private int balance;
	//转账
	void transfer(Account target, int money) {
		//如果无法一次性申请到所有资源就不断尝试，直到申请到
		while(!admin.apply(this, target));
		try {
			//锁住转出账户
			synchronized(this) {
				//锁住转入账户
				synchronized(target) {
					if(this.balance > money) {
						this.balance -= money;
						target.balance += money;
					}
				}
			}
		} finally {
			admin.free(this, target);
		}
	}
}

    # 破坏“不可剥夺”条件

    synchronized关键字无法实现线程自动释放资源，当申请不到资源时，线程就会进入阻塞状态，继续等待；但是 SDK 层面的Lock可以解决这个问题

    # 破坏“循环等待”条件

    要破坏这个条件，首先要先对资源排序，然后按照顺序申请资源。以银行转账为例，有转出账户和转入账户这两种资源，可以以账户 id 作为排序字段，按照账户 id 的大小来申请资源，也就是根据账户 id 的大小来锁定账户 具体实现如下：

    class Account {
	private int id;
	private int balance;
	//转账
	void transfer(Account target, int money) {
		// left：id 小的账户，right：id 大的账户
		Account left = this, right = target;
		if(this.id > right.id) {
			left = target;
			right = this;
		}
		//锁住序号小的账户
		synchronized(left) {
			//锁住序号大的账户
			synchronized(right) {
				if(this.balance > money) {
					this.balance -= money;
					target.balance += money;
				}
			}
		}
	}
}
