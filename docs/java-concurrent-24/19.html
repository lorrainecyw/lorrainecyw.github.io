    19 | CountDownLatch 和 CyclicBarrier ：如何让多线程步调一致

    lorraine 2022-07-05

    本讲以对账系统为例，对账系统的处理逻辑很简单，分为 4 步

    1. 查询未对账订单
    2. 查询派送单
    3. 执行对账
    4. 将差异记录到差异库

    # 用并行优化对账系统

    针对查询未对账订单和查询派送单这两步，可以并行处理，因为这两步并没有先后顺序的依赖。那么并行处理的话，可以让线程 1 执行“查询未对账订单”，线程 2 执行“查询派送单”，等待线程 1 和线程 2 执行完成后，线程 3 执行“对账”和“记录差异”这两步。代码如下：

    while(存在未对账订单){
	// 查询未对账订单
	Thread T1 = new Thread(()->{
		pos = getPOrders();
	});
	T1.start();
	
	// 查询派送单
	Thread T2 = new Thread(()->{
		dos = getDOrders();
	});
	T2.start();
	
	// 等待 T1、T2 结束
	T1.join();
	T2.join();
	
	// 执行对账操作
	diff = check(pos, dos);
	// 差异写入差异库
	save(diff);
}

    但是上面这种代码实现，会重复创建多个线程，这样增加了线程销毁和创建的资源消耗。要进一步优化，就要保证线程复用，而线程复用可以通过线程池来实现。但是上面的主线程是通过join()来等待线程 1 和线程 2 结束，从而知道前两步已经执行完的，而线程池中的线程是不会结束的，那么该如何知道前两步执行完了呢 ？

    # 用 CountDownLatch 实现线程等待

    如何知道前两步已经执行完毕 ？这个问题可以通过计数器解决。设置一个初始值为 2 的计数器，一步执行完就减 1，等到计数器为零就代表前两步的查询操作都执行完了，那么就可以接着执行对账等操作了 在 Java 并发包中，提供了CountDownLatch工具类，它类似于一个计数器，子线程执行完毕后，调用latch.countDown()，计数器就会减 1，而主线程调用latch.await()来等待子线程执行，等到计数器为 0 时，主线程就可以继续执行接下来的操作了 代码实现：

    // 创建 2 个线程的线程池
Executor executor = Executors.newFixedThreadPool(2);
while(存在未对账订单){
	// 计数器初始化为 2
	CountDownLatch latch = new CountDownLatch(2);
	
	// 查询未对账订单
	executor.execute(()-> {
		pos = getPOrders();
		latch.countDown();
	});
	
	// 查询派送单
	executor.execute(()-> {
		dos = getDOrders();
		latch.countDown();
	});
	
	// 等待两个查询操作结束
	latch.await();
	
	// 执行对账操作
	diff = check(pos, dos);
	// 差异写入差异库
	save(diff);
}

    对于上面的代码，如何进一步优化 ？可以看到两个查询操作和对账仍然是串行执行，必须要等待对账和差异记录完毕才可以继续进行下一轮的查询操作，但其实在执行“对账”以及“记录差异”的时候，可以同时开始下一轮的查询操作，并行执行，这样更加节省时间 两次查询操作能够和对账操作并行，对账操作还依赖查询操作的结果，这明显有点生产者 - 消费者的意思，两次查询操作是生产者，对账操作是消费者。既然是生产者 - 消费者模型，那就需要有个队列，来保存生产者生产的数据，而消费者则从这个队列消费数据 具体思路：

    1. 用两个队列来放置查询结果，并且两个队列的元素之间还有对应关系
    2. 订单查询操作将订单查询结果插入订单队列（队列 1），派送单查询操作将派送单插入派送单队列（队列 2），这两个队列的元素之间是有一一对应的关系的
    3. 对账操作可以每次从订单队列取出一个元素，从派送单队列取出一个元素，然后对这两个元素执行对账操作，这样数据一定不会乱掉

    思路实现：

    1. 线程 1 执行订单的查询工作，线程 2 执行派送单的查询工作
    2. 当线程 1 和 2 都各自生产完 1 条数据的时候，通知线程 3 执行对账操作
    3. 必须要保证线程 1 和线程 2 的步调一致，不能一个跑得太快，一个跑得太慢，得相互等待。只有这样才能做到各自生产完 1 条数据的时候，通知线程 3

    # 用 CyclicBarrier 实现线程同步

    针对上面的优化思路，可以通过 Java 并发包的CyclicBarrier来实现。CyclicBarrier类似于计数器，并且这个计数器是可重用的。在创建CyclicBarrier的时候，会传入了一个回调函数，当计数器减到 0 的时候，会调用这个回调函数 具体做法：

    1. 创建初始值为 2 的CyclicBarrier
    2. 线程 1 负责查询订单，当查出一条时，调用barrier.await()来将计数器减 1，同时等待计数器变成 0
    3. 线程 2 负责查询派送单，当查出一条时，也调用barrier.await()来将计数器减 1，也等待计数器变成 0
    4. 当 T1 和 T2 都调用barrier.await()的时候，计数器会减到 0，此时 T1 和 T2 就可以执行下一条语句了，同时会调用 barrier 的回调函数来执行对账操作，需要另外开一个线程来执行。而计数器会自动重置为 2

    回调函数需要另外开一个线程执行的原因： CyclicBarrier的回调函数执行在一个回合里最后执行await()的线程上，而且同步调用回调函数check()，调用完check()之后，才会开始第二回合。所以check()如果不另开一线程异步执行，就起不到性能优化的作用了 另外开一个线程执行的做法： 通过线程池来完成，实现线程复用，但是线程池里只能有一个线程，是为了防止多线程而导致数据取错了（线程 A 和线程 B 同时执行，线程 A 先取了订单 1 ，而线程 B 先取了派送单 1，这样 A 就取到了 派送单 2，导致订单与派送单不匹配）

    // 订单队列
Vector<P> pos;
// 派送单队列
Vector<D> dos;
// 执行回调的线程池
Executor executor = Executors.newFixedThreadPool(1);

final CyclicBarrier barrier = new CyclicBarrier(2, ()->{
	executor.execute(()->check());
});

void check(){
	P p = pos.remove(0);
	D d = dos.remove(0);
	// 执行对账操作
	diff = check(p, d);
	// 差异写入差异库
	save(diff);
}

void checkAll(){
	
	// 循环查询订单库
	Thread T1 = new Thread(()->{
		while(存在未对账订单){
			// 查询订单库
			pos.add(getPOrders());
			// 等待
			barrier.await();
		}
	});
	T1.start(); 
	
	// 循环查询运单库
	Thread T2 = new Thread(()->{
		while(存在未对账订单){
			// 查询运单库
			dos.add(getDOrders());
			// 等待
			barrier.await();
		}
	});
	T2.start();
}

    # CountDownLatch 和 CyclicBarrier 的区别

    1. CountDownLatch主要用来解决一个线程等待多个线程的场景，可以类比旅游团团长要等待所有的游客到齐才能去下一个景点
    2. CyclicBarrier是一组线程之间互相等待，更像是几个驴友之间不离不弃
    3. CountDownLatch的计数器是不能循环利用的，也就是说一旦计数器减到 0，再有线程调用await()，该线程会直接通过。但CyclicBarrier的计数器是可以循环利用的，而且具备自动重置的功能，一旦计数器减到 0 会自动重置到你设置的初始值。
    4. CyclicBarrier可以设置回调函数
