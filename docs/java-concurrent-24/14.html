    14 | Lock 和 Condition（上）：隐藏在并发包中的管程

    lorraine 2022-07-05

    并发编程领域有两大核心问题：互斥和并发。Java SDK 并发包通过 Lock 和 Condition 两大接口来实现管程，Lock 用于解决互斥问题，让同一时刻只有一个线程能访问，Condition 用于解决同步问题

    # 再造管程的理由（为什么要编写 Lock）

    Java 语言本身提供的synchronized就是管程的一种实现，为什么要再编写 Lock 接口 ？可能是性能原因吗 ？不是的。虽然在JDK 1.5及之前，synchronized性能不如 SDK 中的 Lock，但是在JDK 1.6的时候对synchronized做了很多优化，性能已经追上来了，所以不是性能的原因 synchronized有一个弊端，那就是在申请锁的时候，如果获取不到，线程就直接进入阻塞状态了，并且在获得锁之前无法被中断（获取锁后会响应中断），这样线程什么也干不了，自然也无法释放自己已经获取到的其他锁资源。如果造成了死锁，那么“不可剥夺”条件就无法被打破 而 Lock 接口创建了 3 个方法来解决这个问题，让线程可以自动释放自己获得的资源，来破坏“不可剥夺”条件

    1. lockInterruptibly()：支持响应中断。让处于阻塞状态的线程可以响应中断信号，这样线程就能够被唤醒，就有机会释放自己持有的其他锁
    2. tryLock(long Time, TimeUnit unit)：支持超时。尝试获取锁，在一定的时间里如果没有获得锁，那么线程不会进入阻塞状态，而是返回一个错误，这样线程就有机会释放自己持有的其他锁
    3. tryLock()：非阻塞式获取锁。尝试获取锁，如果没有获取到锁，直接返回 false，不阻塞，这样就有机会释放自己持有的其他锁

    # 如何保证可见性

    第 2 讲中讲述了可见性是通过Happens-Before规则来保证的，synchronized能保证可见性也是因为Happens-Before中管程锁的规则，那么Lock是如何保证的呢 ？

    class X {
	 private final Lock rtl = new ReentrantLock();
	 int value;
	 public void addOne() {
		 // 获取锁
		 rtl.lock(); 
		 try {
			 value+=1;
		 } finally {
			 // 保证锁能释放
			 rtl.unlock();
		 }
	 }
}

    如果线程 A 执行了addOne()，那么value += 1后 value 的值对后来的线程 B 可见吗 ？ Lock 保证可见性是利用了Happens-Before规则中与volatile有关的规则（对volatile变量的写操作Happens-Before对于volatile变量的读操作）。ReentrantLock内部持有一个volatile的成员变量state，加锁lock()时将state加一，解锁unlock()时将state减一，也就是说在加锁和解锁的时候都会读写state的值。根据相关的Happens-Before规则推断：

    1. 顺序性规则：对于线程 A，value += 1的操作Happens-Before于解锁unlock()
    2. volatile变量的规则：线程 A 解锁unlock()中对state的写Happens-Before于线程 B 加锁lock()中对state的读，简而言之，线程 A 的unlock()操作Happens-Before于线程 B 的lock()操作
    3. 传递性规则：线程 A 的value += 1操作Happens-Before于线程 B 的lock()操作

    从上面的推断，可以看出value += 1后 value 的值对后来的线程 B 可见吗

    # 可重入锁

    ReentrantLock和synchronized都是可重入锁。Reentrant本身翻译过来就是“可重入”的意思。可重入锁：获得锁的线程可以重复获取同一把锁，获取多少次就要释放多少次。可重入锁在重复获取同一把锁的过程中能够继续执行，而不可重入锁如果重复获取，线程会进入阻塞状态 除了可重入锁外，还有可重入函数。可重入函数：多个线程可以同时调用该函数，并且都获得正确的结果，同时在一个线程内支持线程切换，并且依旧保证结果正确。也就是说，可重入函数是线程安全的

    # 公平锁与非公平锁

    ReentrantLock有两个构造函数，一个是无参构造函数，默认创建非公平锁，另一个需要传入 fair 参数，根据 fair 参数来判断是否创建公平锁。公平锁：根据线程的等待时间来唤醒线程，哪个线程等待的时间长就唤醒哪个线程。非公平锁：随机唤醒线程。synchronized采用的就是非公平锁 非公平锁的性能由于公平锁。但如果锁的持有时间较长或者请求锁的平均时间间隔较长，还是采用公平锁较好，以免一些等待时间长的线程一直无法获得锁

    # 用锁的最佳实践

    出自并发大师 Doug Lea《Java 并发编程：设计原则与模式》一书

    1. 永远只在更新对象的成员变量时加锁
    2. 永远只在访问可变的成员变量时加锁
    3. 永远不在调用其他对象的方法时加锁

    为什么要遵守第 3 条 ？ 因为调用其他对象的方法，实在是太不安全了，也许“其他”方法里面有线程sleep()的调用，也可能会有奇慢无比的 I / O 操作，这些都会严重影响性能。更可怕的是，“其他”类的方法可能也会加锁，然后双重加锁就可能导致死锁
