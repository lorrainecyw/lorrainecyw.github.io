    # 如何解决原子性问题

    原子性：所有操作要么全部执行，要么全部不执行。所以在同一时刻只能有一个线程执行 从 第一讲 (opens new window) 可以知道，原子性问题的源头是线程切换，那么要解决原子性问题就可以通过禁用线程切换来解决，而线程切换依赖于 CPU 中断，那么禁止 CPU 中断就可以解决原子性问题 在单核时代，确实是可行的，禁止 CPU 中断可以保证获得 CPU 使用权的线程可以不断执行直到执行完成，就不会出现原子性问题。但是在多核时代，由于有多个 CPU，此时禁止 CPU 中断，只能够保证“每个 CPU 上只有一个线程在执行”，并不能保证此刻只有一个线程在执行

    “同一时刻只有一个线程执行”这个条件非常重要，成为“互斥”，只要保证了对共享变量的修改是互斥的，那么就能够保证原子性

    说到互斥，肯定会想到“锁”。先确定好共享资源（锁要保护的资源），创建保护资源的锁，在进入临界区（需要互斥执行的代码）前，先加锁，执行完临界区的代码后，释放锁。以上就是互斥锁使用的简单介绍，接下来看 java 提供的锁的一种实现：synchronized关键字

    # synchronized 关键字

    # 1. synchronized 的用法

    在使用synchronized时，Java 会默默在synchronized修饰的方法或者代码块前后分别加上加锁和解锁的操作，这样能够保证加锁和解锁这两个操作是成对出现的，不用担心忘记解锁 synchronized有三种用法：

    1. 修饰非静态方法：锁定的是当前实例对象 this
    2. 修饰静态方法：锁定的是当前类的 class 对象，因为静态方法是属于类的
    3. 修饰代码块：既可以锁定类的 class 对象，也可以锁定实例对象
    class Test {
	//修饰非静态方法，相当于 synchronized(this) void say()
	synchronized void say() {
		//临界区
	}
	
	//修饰静态方法，相当于 synchronized(Test.class) void write()
	synchronized void write() {
		//临界区
	}
	
	Object obj = new Object();
	void read() {
		//修饰代码块
		synchronized(obj) {
			//临界区
		}
	}
}

    # 2. 例子：用 synchronized 解决 count += 1 问题

    为了保证count += 1语句的原子性，那么就要使实现这个操作的方法被synchronized修饰，如下：

    class ClassRoom {
	long count = 0L;
	long get() {
		return count;
	}
	synchronized long addOne() {
		count += 1;
	}
}

    那么就能够保证同一时刻只有一个线程执行addOne()，保证了原子性，并且通过Happens-Before规则的“管程中锁的规则”（对一个锁的解锁Happens-Before于后续对这个锁的加锁），所以解锁前做的操作对后续加锁是可见的，保证了可见性，那么执行了多少次addOne()，count 就会增加多少 但是，虽然保证了 count 的值不会有问题，但是 count 的值对get()是可见的吗？这是没法保证的。为了实现 count 的值对get()可见，那么可以在get()上也加上synchronized，这样get()和addOne()加的都是当前实例对象 this 的锁，是同一个锁，那么就能保证可见性了。 修改后的代码如下：

    class ClassRoom {
	long count = 0L;
	synchronized long get() {
		return count;
	}
	synchronized long addOne() {
		count += 1;
	}
}
