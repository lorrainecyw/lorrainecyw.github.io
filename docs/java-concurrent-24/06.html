    06 | 用“等待-通知”机制优化循环等待

    lorraine 2022-07-03

    在并发量不大的情况下，可能循环几次就能够得到锁，这个时候循环等待是可用的，不会太消耗 CPU；但是在并发量大的情况下，可能要循环上千次甚至上万次才能获得锁，在这个循环的过程中，会一直消耗 CPU，导致 CPU 的利用率达到比较高的程度，性能会较低，所以说，要用“等待 - 通知”机制来优化循环等待

    # “等待 - 通知”机制的过程描述

    一个完整的“等待 - 通知”机制主要分为以下几步：

    1. 线程尝试获取互斥锁
    2. 线程获取到锁后，进入临界区
    3. 如果线程需要的资源不满足时，释放互斥锁，进入等待状态
    4. 当线程需要的资源满足时，通知在等待的线程，线程重新获取互斥锁

    # 用 synchronized 实现“等待 - 通知”机制

    在 Java 中，实现“等待 - 通知”机制的方式有多种，其中一种实现方式就是通过synchronized配合wait()、notify()、notifyAll()来实现的 什么时候需要等待呢？条件不满足的时候，调用wait()让线程阻塞，进入等待队列进行等待 什么时候需要唤醒呢？条件满足的时候，调用notify()或notifyAll()唤醒线程，告诉它条件曾经满足过

    //等待
synchronized(xxx) {
	if(条件不满足) {
		wait();
	}
}

//唤醒
synchronized(xxx) {
	if(条件满足) {
		notify();	//或者notifyAll()
	}
}

    可以看出，wait()、notify()、notifyAll()的调用都是在获得锁的情况下，并且获得的是哪个对象的锁，调用的就是哪个对象的方法，例如获得的是 this 对象的锁，那么调用的就是this.wait()。如果在synchronized(xxx) {}外部调用wait()等方法，就会抛出一个运行时异常：java.lang.IllegalMonitorStateException。此外，wait()、notify()、notifyAll()都是 Object 拥有的方法

    # 例子：银行转账问题

    在银行转账问题上，原本采用的是循环等待，现在采用“等待 - 通知”机制，需要考虑 4 个因素：

    1. 互斥锁：由于Administrator是单例的，所以可以用Administrator对象所谓互斥锁
    2. 线程要求的条件：转入账户和转出账户资源都没有被占用
    3. 什么时候等待：线程要求的条件不满足的时候等待
    4. 什么时候通知：有线程释放资源时就通知

    具体实现如下：

    //管理员
class Administrator {
	//存放已经被占用的资源
	private List<Object> sourses = new ArrayList<>();
	synchronized void apply(Object from, Object to) {
		//如果 from 或者 to 有一个被占用了，那就无法申请到
		while(sources.contains(from) || sources.contains(to)) {
			try {
				wait();
			} catch (Exception e) {
			
			}
		}
		
		sources.add(from);
		sources.add(to);
	}
	synchronized void free(Object from, Object to) {
		sources.remove(from);
		sources.remove(to);
		notifyAll();
	}
}

    为什么上面的例子用的是notifyAll() ? notifyAll()：通知等待队列中的所有线程，notify()：随机通知等待队列中的一个线程。虽然最终只会有一个线程获得锁，使用notifyAll()看起来增加了锁竞争的风险，但是使用notify()是有风险的。由于notify()是随机通知等待队列中的一个线程，那么可能导致某些线程永远都没有被通知到 例如线程 1 占有了 A、B 这两个资源，线程 2 占有了 C、D 这两个资源，线程 3 同样需要的是 A、B，线程 4 需要的是 C、D，当线程 1 释放了 A、B 时，notify()唤醒的却是线程 4，线程 4 并不需要这两个资源，而线程 3 却会面临可能不被唤醒的情况
