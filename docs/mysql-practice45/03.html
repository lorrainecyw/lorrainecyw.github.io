    03 | 事务隔离：为什么你改了我还看不见？

    lorraine 2022-06-03 MySQL事务

    # 事务简单介绍

    什么是事务？ 事务是并发控制的单位，是用户定义的一个操作序列。事务保证一组数据库操作要么全部执行成功，要么全部执行失败。事务由引擎层来支持（MyISAM 不支持事务，InnoDB 支持事务） 事务的特性：

    • 原子性（Atomic）：所有的操作要么全部执行成功，要么全部执行失败
    • 一致性（Consistency）
    • 隔离性（Isolation）
    • 持久性（Durability）：事务进行的修改是永久的

    # 事务隔离级别

    当事务同时执行时，就可能出现 脏读dirty read、不可重复读non-repeatable read、**幻读phantom read**等问题，为了解决这些问题，就有了“隔离级别”的概念 隔离得越严实 -> 效率越低，所以需要根据业务寻找一个平衡点 隔离级别总共有 4 种（MySQL 默认隔离级别为RR，Oracle 默认隔离级别为RC）：

    1. 未提交读（RU，Read Uncommitted）：其他事务可以看到未提交事务所做的修改
    2. 提交读（RC，Read Committed）：事务只能看到已提交事务所做的修改
    3. 可重复读（RR，Repeatable Read）：一个事务执行过程中看到的数据总是和事务启动时看到的一致
    4. 可串行化（Serializable）：对于同一行记录，写需要加写锁，读需要加读锁。当发生读写锁冲突时，后访问的事务必须等前一事务执行完成，才能继续执行

    在实现上，数据库里面会创建一张 视图（read-view）。在RR级别下，视图是在事务启动的时候就创建好的，在整个事务期间都使用同一张视图；在RC级别下，视图是在每个 SQL 语句开始执行的时候创建的 通过设置transaction-isolation参数来修改隔离级别，并通过show variables like 参数命令来查看

    # 事务隔离的实现

    通过 undolog（回滚日志）实现

    实际上，MySQL 中每条记录更新的时候都会同时记录一条回滚操作到undolog中。每行记录都可以通过回滚操作，得到之前状态的值，也就是同一条记录在系统中存在多个版本，这就是MVCC（多版本并发控制） 例如：某行记录的 c 字段从 1 被改成 2、3、4，那么回滚日志的记录如下： 不同时刻启动的事务会有不同的视图（read-view），如果想要得到 c = 1 的视图read-view A，那么就需要经历三次回滚操作的执行 **回滚日志会在什么时候删除？**自然是在没有比当前回滚日志更早的视图的时候（不需要的时候），就可以删除了。由此也可能，如果一直有比回滚日志更早的视图，那么当前回滚日志就不能删除，那么回滚日志积累得越来越多，会占用大量的存储空间。所以 尽量不要使用长事务 长事务的缺点：

    1. 因为长事务意味着系统里会存在很多很老的事务视图，老视图越多，回滚日志就越多，会占用大量存储空间
    2. 在MySQL 5.5 之前，undolog 和 数据字典一起放在 ibdata 文件中，所以即使长事务最终被提交，也只是 ibdata 文件里的回滚段被清理了，文件大小并不会发生改变，仍然占用空间
    3. 长事务还占用锁资源等，可能会拖垮整个库

    # 事务启动方式

    事务启动方式有两种：

    1. 显式启动，使用begin或start transaction开启事务，使用commit提交事务，使用rollback回滚
    2. set autocommit = 0，关闭当前线程自动提交。也就是执行语句之后，事务就开启了，但是并不会自动提交，直到执行了commit或者rollback，或者是断开连接，这就很容易导致长事务

    如何避免长事务？

    1. 通过显式语句的方式启动事务或者set autocommit = 1，让事务能够自动提交
    2. 使用commit work and chain来提交事务并开启下一事务。由于执行commit提交上一个事务，然后在begin启动下一个事务总共要两次交互，所以使用commit work and chain可以减少交互次数，减少多一条语句的开销
